# GRPO 训练显存分析 & OOM 调试记录

## 一、OOM 发生在哪个阶段

GRPO 一个 step 的流程：`rollout(vllm generate)` → `reward` → `compute_log_probs(actor forward)` → `compute_ref_log_probs(ref forward)` → `compute_advantage` → `update_actor(forward+backward)` → `save_checkpoint`

OOM 先后出现在两个阶段：
1. **`compute_log_probs`**（#6～#13）——`@torch.no_grad()`，无梯度
2. **`update_actor`**（#13 解决后转移过来）——有梯度 + gradient checkpointing，显存压力更大

**两者根因相同：PyTorch CUDA allocator 碎片累积**——每次 micro_batch forward 后，all-gather 产生的临时 tensor 没有立即归还，逐步积累到 OOM。`expandable_segments` 只解决 segment 粒度碎片，不解决这个。

**根本解法**：micro_batch 循环内加 `torch.cuda.empty_cache()`（见三、修复）。

## 二、显存构成（理论分析）

代码入口（`fsdp_workers.py:609`）：
```python
def compute_log_probs(self, data: DataProto):
    data = data.to(torch.cuda.current_device())  # 整个 batch data 搬上 GPU
    output = self.actor.compute_log_prob(data=data)  # 按 micro_batch 切分 forward
```

**显存占用 = 常驻部分 + data 部分 + forward 峰值**

### 常驻部分

| 项目 | 每卡显存 | 说明 |
|------|---------|------|
| bf16 活跃参数（FSDP 分片） | ~1 GB | 3.75B × 2B / 8 卡 |
| vllm 残留（offload 后） | ~3.7 GB | 实测 vllm offload 后 4.68 GB，含参数 ~1 GB |
| fp32 优化器状态（m+v+master） | ~5.7 GB | 3.75B × 4B × 3 / 8 卡；**Adam 懒初始化，第一次 step() 后才分配，且 compute_log_probs 阶段 offload 在 CPU** |
| **compute_log_probs 开始时合计** | **~4.68 GB** | 实测值（Step 1）；Step 2 起涨到 ~7.93 GB（optimizer state 初始化后的永久增量） |

### data 部分（搬上 GPU 的完整 batch 数据）

`data.to(cuda)` 把该卡负责的所有样本一次性搬上 GPU，包含：
- `input_ids / attention_mask / position_ids / responses`（int64 token 数据）
- `multi_modal_inputs`（图片 pixel values，**大头**）

每条样本图片大小：最大 `max_pixels=1M` × fp32 = 4 MB

| 配置 | 每卡 rollout 条数 | 图片 data 估算 |
|------|----------------|--------------|
| batch=512，n=8，8 卡 | 512×8/8 = 512 条 | 512 × 4 MB ≈ 2 GB |
| batch=128，n=8，8 卡 | 128×8/8 = 128 条 | 128 × 4 MB ≈ 0.5 GB |

> 实际 data 占用比上表大——序列可能很长。实测中 in_use 数字包含 data，具体大小难以单独隔离。

### forward 峰值（每个 micro_batch 的临时显存）

FSDP forward 时每层做 all-gather，临时重建完整参数（~7.5 GB），加上 logits + 激活：

| micro_batch | total_nnz (unpadded) | logits (nnz×152064×bf16) | 中间激活 | 峰值合计 |
|-------------|---------------------|--------------------------|---------|---------|
| 32 | ~96K tokens | ~28 GB | ~15 GB | ~43 GB |
| 16 | ~48K tokens | ~14 GB | ~8 GB | ~22 GB |
| 8 | ~24K tokens | ~7 GB | ~4 GB | ~11 GB |
| 4 | ~12K tokens | ~3.5 GB | ~2 GB | ~13 GB（含 all-gather） |

vocab_size = 152,064 是 logits 显存大的原因。

## 三、实测显存时间线（#14）

### 初始化阶段

```
HuggingFace model init:   1.30 GB    ← actor 模型权重加载到 GPU（bf16）
FSDP module init:         3.72 GB    ← FSDP 封装 actor，分配梯度缓冲区等
HuggingFace model init:   3.72 GB    ← ref 模型加载（共享 actor 权重，无额外显存）
FSDP module init:         7.04 GB    ← FSDP 封装 ref 模型
Optimizer init:           7.04 GB    ← Adam 对象创建（懒初始化，暂不分配 state）
vllm init:                4.62 GB    ← actor/ref FSDP offload 到 CPU；vllm 框架初始化
```

### Rollout 阶段（每次 step 前的 weight sync）

| 日志行 | Step 1 | Step 2 | 变化 | 在做什么 |
|--------|--------|--------|------|---------|
| Before state_dict() | 4.62 GB | 7.93 GB | Step2 高 **+3.3 GB** | vllm sleep 后；Step 2 起多了 optimizer state（第一次 step() 后懒初始化的永久增量） |
| After state_dict() | 8.78 GB | 12.45 GB | +4.16 GB | FSDP all-gather 聚合分片，提取为临时 state_dict |
| After sync weights | 60.95 GB | 64.62 GB | +52 GB | 权重复制进 vllm + 预分配 KV cache 池（~52 GB = 模型权重 + KV cache） |
| After del state_dict | 56.79 GB | 60.10 GB | −4 GB | 删临时 state_dict，vllm 参数副本保留 |
| Before vllm offload | 71.04 GB | — | +14 GB | 推理完成，KV cache 实际填充 ~14 GB |
| After vllm offload | **4.68 GB** | — | −66 GB | vllm 完全释放 → **compute_log_probs 起点** |

### compute_log_probs 阶段：OOM 现象（#12b，修复前）

配置：batch=128，n=8，8 卡 → 每卡 128 条 rollout → micro_batch=4 → 32 次迭代

```
开始时：~5 GB
进度 0/32 → 12/32：正常，每批 ~10s
进度 13/32：💥 OOM
  Tried to allocate:   23.18 GiB
  GPU free:            22.85 GiB   ← 差 0.33 GiB！
  In use:              56.26 GiB   ← 从 ~5 GB 涨到 56 GB（13 批 × 3.9 GB/批）
  PyTorch allocated:   91.57 GiB
  Reserved unused:     15.49 GiB   ← 碎片
```

**碎片速率**：(56 − 5) / 13 ≈ **3.9 GB/micro_batch**

### 修复：empty_cache + .cpu()

**文件**：`verl/workers/actor/dp_actor.py`（compute_log_probs 循环内）

```python
# 修复前
log_probs_lst.append(log_probs)

# 修复后
log_probs_lst.append(log_probs.cpu())   # 结果移到 CPU
torch.cuda.empty_cache()                # 清除 allocator 碎片
```

同理，`update_actor` 的 `loss.backward()` 后也加了 `empty_cache()`。

**效果**：每次 forward 后 GPU 回落到 ~5 GB，峰值仅 ~28 GB，**64/64 全部通过**。降 micro_batch/batch_size 是治标，empty_cache 才是根本解法——碎片按次数累积，不论 micro_batch 多小，64 次后照样 OOM。

### update_actor 阶段 OOM（#13 遇到，#14 修复）

compute_log_probs 通过后，OOM 转移到 `update_actor`（forward + backward，有梯度）：

```
Tried to allocate:  38.88 GiB
Free:               13.47 GiB
In use:             65.64 GiB
位置：attention softmax
```

原因：update 有梯度 + gradient checkpointing，激活值需保留；`micro_batch_for_update=8`（yaml 默认值）；update 循环内无 `empty_cache()`。**修复同上**，`micro_batch_for_update` 降到 2。

### Step 1 全部通过（#14 ✅）

```
compute_log_probs:     64/64 ✅
compute_ref_log_probs: 64/64 ✅
update_policy:         64/64 ✅
checkpoint saved:      ✅
```

## 五、论文配置 vs 实际差异

| 参数 | 论文 | 我们的 yaml | 差异影响 |
|------|------|-----------|---------|
| fsdp_size | 8 | 8 | ✅ 一致 |
| global_batch_size | 512 | 512 | ✅ 一致 |
| micro_batch (experience) | **32** | 64（默认） | 2x，forward 显存翻倍 |
| max_response_length | **4096** | 8192（默认） | 2x，token 数翻倍 |
| gpu_memory_utilization | **0.75** | 0.75 | ✅ 一致 |
| PPO epochs | **20** | 1 | 影响训练效果 |
| VAS update freq | **56** | 4 | 更新频率差异大 |

**论文用 8 卡 A100-80GB + batch=512 + micro=32 + max_resp=4096 能跑通**，但我们同样配置（#9）OOM 122 GiB。可能原因：
1. 论文用了 H100/H800
2. 显存泄漏是代码 bug，论文版本可能已修复
3. 图片 multi_modal_inputs 实际大小差异

## 六、可能的进一步优化

1. **CPU offload**：`worker.actor.fsdp.enable_cpu_offload=true` — 优化器状态放 CPU，省 ~5.7 GB/卡
2. **gradient checkpointing**：已开启（`enable_gradient_checkpointing: true`）
3. **减小图片分辨率**：`max_pixels` 调小，省 multi_modal_inputs 显存
4. **activation offload**：PyTorch FSDP 支持，verl 是否已接入待确认
5. **混合精度检查**：确保 forward 全程 bf16，不意外升 fp32

## 七、显存诊断工具

### torch.cuda.memory._dump_snapshot（精确定位碎片来源）

参考：[Awesome-ML-SYS-Tutorial: mem-snapshot](https://github.com/zhaochenyang20/Awesome-ML-SYS-Tutorial/blob/main/torch/mem-snapshot/readme.md)

```python
torch.cuda.memory._record_memory_history(max_entries=100000)
# ... 训练代码 ...
torch.cuda.memory._dump_snapshot("mem_snapshot.pkl")
torch.cuda.memory._record_memory_history(enabled=None)
```

生成的 `.pkl` 上传到 [pytorch.org/memory_viz](https://pytorch.org/memory_viz) 可视化：
- **Active Memory Timeline**：每个 tensor 的分配/释放时间线，精确定位哪个 op 产生碎片
- **Allocator State History**：白色块 = 碎片（已预留但未分配）

社区在 VLM 训练中发现：**显存碎片主要源于 Qwen VL fast tokenizer 的 image processor 阶段**，与本项目每 micro_batch 泄漏 ~4 GB 的现象吻合。empty_cache 后仍 OOM 时用此工具精确定位。
